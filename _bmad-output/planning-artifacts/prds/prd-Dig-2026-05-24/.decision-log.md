## Decisions

### D-1: Stakes calibration
**Hobby / passion project.** Brief explicitly states "passion project first, learning project second, portfolio piece third." Section depth calibrated accordingly — complete but not enterprise-weight.

### D-2: Visual aesthetic
**Dark UI, data-forward. Obsidian × Spotify.** Night-sky graph canvas (dark background, glowing nodes). Clean typography, no clutter. Anti-reference: SaaS dashboards. Captured in PRD §Aesthetic and Tone.

### D-3: Search behavior
**Live autocomplete (Option A).** User sees results appearing as they type, clicks to confirm. MusicBrainz artist search API as autocomplete source. Captured in FR-1.

### D-4: Shareable URLs
**In scope for v1.** `/artist/[slug]` route is not a social feature — it is baseline web behavior and essential for sharing discoveries. Low effort, high value. Captured in §4.6 and §6.1.

### D-5: Graph directionality layout
**Spatial separation (Option B).** Left = upstream influences (roots), right = downstream influences (legacy), Focal Artist at center. Layout carries directional meaning; color reinforces. No legend required. Captured in FR-4 and Addendum.

### D-6: Data source architecture
**MusicBrainz ruled out as influence source.** Pre-build sampling confirmed zero influence edges for all five test artists. Wikipedia infobox (primary upstream) + Wikidata P737 SPARQL (secondary upstream / primary downstream via reverse query) is the corrected hierarchy. MusicBrainz retained for metadata only. Full details in `addendum.md`.

### D-7: Spotify preview abstraction
**Preview integration must be behind a swappable abstraction.** Spotify's API fragility is a known risk. Captured in FR-15 (NFR) and `addendum.md`.
